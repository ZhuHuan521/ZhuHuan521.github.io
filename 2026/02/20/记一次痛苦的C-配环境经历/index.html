前言

C++ 名不虚传的难，之前多次尝试用vscode写 C++ ，因为喜欢轻量的东西，然而屡战屡败。

第一次是要include一个WinRT，怎么都include不进，然后转用vs，vcpkg一键安装，成功完成，配合使用这个桌面歌词软件。嗯对，就是我现在正在用的这个。

然后第二次用vscode写，就是调试总跑不起来 (现在知道好像它自带的这个调试不会自动链接所有的源文件，晕死) 然后把写好的代码又丢到vs里面去了，成功。

接下来就是这次了，这次我说什么都要用上vscode！

进入正题

先说我自己的环境吧：

  • 系统：Windows 11 24H2 (史)
  • vscode版本：1.109.5 (user setup)
  • mingw: gcc version 15.2.0 (MinGW-W64 x86_64-ucrt-posix-seh, built by Brecht Sanders, r1)
  • CMake / CMake Tools vscode内下载

由于整个过程踩的坑实在是太多太多了，我都不太记得清楚了，但是可以根据与gemini的对话记录回忆起来。

初出茅庐 MINGW的尝试： 与"tasks.json"的邂逅

按着网上一搜一大把的教程安装了mingw，复制了一个tasks.json文件，拿来调试。Hello World一遍跑通。然而我似乎只能跑通它。按照我的习惯，也是工程开发中所提倡的，把.hpp,.cpp诸如此类的文件分开来存。

第一个问题出现了：找不到.hpp文件了！

对，我试了很多方法，比如说先把.hpp和.cpp放到一起，然后直接 <"a.hpp"> 没用，在includepath里添加也没用，甚至写绝对路径都找不到，逆天了孩子们。

最后重启vscode好了。。。

更开心的要来了：引入libcurl库

这下是没招了，按着网上给的tasks.json写了，毫无卵用，与gemini一番对话下来，越搞越乱。。。而且如果后面引入的库越来越多，每引用一个就要改一下tasks.json，简直是噩梦。

新发现 CMake破局： CMakelists.txt的配置

正当“山重水复疑无路”的时候，cnblogs上一篇使用CMake进行c++开发的文章极大的引起了我的兴趣，经过相关了解之后知道它居然如此好用！一个 CMakelists.txt 搞定一切！（貌似也不是一切，毕竟现在还有点小毛病，后面再说）

没错，cmake是对的，它还兼容vcpkg，只需要

就如此简单的引入了！别忘了配置好vcpkg的目录哦

如此看来，CMake是不是就一劳永逸了？

其实我自己在做的时候也是碰到了不少问题的，比如说vcpkg默认安装的msvc版本的库，mingw不能用这种问题。话说为什么当时安装，msvc版名字里面不标“msvc”啊，只有mingw版会标“mingw”。

然后就是右上角那个愚蠢的调试按钮，它只会按照tasks.json里的去呆板地执行，具体怎么让它redirect到CMake是个谜，我认为肯定是可以的，网上也有相关文档，我照做了，但没用。问了gemini，它说弄不了，并且极力建议我不要用那玩意，会变得不幸。

实际上确实是这样的，vscode的代码提示叫IntelliSense，根本不Intell (智慧) 好吧！我一按右上角那玩意，结果因为tasks.json是错的，然后提交到gdb里面报错，然后IntelliSense就开始给你画波浪线了😈😈😈

所以我现在都还是用左下角的“播放”和“bug”图标，目前没啥问题了。

总结：自己菜 不过C++确实阴

目前，开发软件，引入需要的库是十分优雅的，直接在vcpkg里面安装，到时候别的项目需要使用也可以直接复用vcpkg目录里的东西，不会重复造轮子。

但仍然存在的问题是能不能做一个tasks.json指向CMake，CMake Tools可以生成IntelliSense的配置，我觉得是可以生成tasks.json的，应该是我还没发现。

此外，这个调试我还是搞不懂怎么搞，变量监视啥的没用。

累了，写了这么多，等上面的问题解决了又能水一篇()
